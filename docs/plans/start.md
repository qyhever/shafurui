# PC 视频相册页面方案

## 概述

目标是做一个面向 PC Web 的视频相册页，页面从上到下展示视频列表，并按日期时间分组。当前视频来源链路是：

1. 手机导出视频到本地目录
2. 使用脚本按拍摄日期整理到中间目录
3. 使用脚本把 `mov` / `m4v` 无损重新封装为 Web 更友好的 `mp4`
4. 使用脚本为缺失封面的视频生成同名 `jpg`
5. 使用 `rsync` 上传整理后的目录到远程服务器
6. 视频文件和封面图落到 nginx 的 `html` 或独立静态目录
7. 提供一个 Go API 按配置扫描该目录并返回视频列表
8. 前端页面调用 API 渲染相册

这个方案整体可行，适合私有自用场景。推荐采用“shell 脚本负责离线预处理，nginx 提供静态视频和封面文件，Go 只负责目录扫描和列表接口，前端负责分组展示”的分层方式。手机导出的 `mov` 建议保留原文件，同时在中间目录生成 Web 播放用的 `mp4`，不要直接覆盖原始素材。

## 可行性分析

### 方案为什么可行

- 视频文件直接由 nginx 提供静态访问，天然支持 `Range` 请求，适合浏览器视频播放。
- Go 读取本地目录生成 JSON 列表，实现简单，部署成本低。
- 相册页只需要消费结构化数据，不直接参与文件系统逻辑，前后端职责清晰。
- 你现在是私有自用场景，不需要一开始就引入数据库、对象存储或复杂媒体服务。

### 当前链路的优点

- 简单直接，维护成本低。
- `rsync` 很适合同步预处理后的文件。
- 日期分组和封面生成放在上传前处理，可以减少服务器运行时 CPU 和 I/O 压力。
- 静态文件和 API 分离后，后续优化空间大。

### 当前链路的主要风险

- 如果每次请求都实时扫描目录，视频数量大了以后接口会变慢。
- 如果按“视频元数据时间”分组，需要额外读取容器元数据，仅靠文件名或文件时间不一定准确。
- 手机导出后的文件命名、目录层级、时间戳保留方式如果不稳定，会影响分组准确性。
- 如果预处理脚本没有稳定执行，新增视频可能缺少封面图或落在兜底日期目录下。

## 推荐架构

### 结构分层

- nginx：
  - 提供视频静态访问
  - 处理浏览器视频播放、缓存、`Range`
- Go API：
  - 读取配置的视频目录并扫描
  - 提取视频元数据
  - 生成按时间排序后的结构化列表
  - 使用配置的视频访问前缀返回视频 URL 和同名封面 URL
- PC Web：
  - 按日期分组展示
  - 支持视频预览和播放

### 推荐目录约定

建议不要把视频直接混放在 nginx 默认站点根目录零散位置，最好单独规划：

```text
/data/www/site/
  index.html
  assets/
  videos/
    2026-06-30/
      a.mp4
      a.jpg
    2026-07-01/
      b.mp4
      b.jpg
```

这样做的好处：

- 目录含义清晰
- 后续备份方便
- Go API 的扫描范围明确
- nginx location 配置更容易隔离

## 时间分组策略

手机导出后的视频目录通常是扁平的，因此在上传前先用脚本预处理。脚本负责把视频复制到按日期分组的中间目录，Go API 仍保留元数据读取和兜底排序能力。

### 推荐时间优先级

1. 视频元数据中的创建时间或拍摄时间
2. 文件名中的时间
3. 文件 `mtime`

### 为什么不能只靠 `mtime`

- `rsync` 配置不当时，文件时间可能变成上传时间。
- 即使保留了时间，`mtime` 也不一定等于真实拍摄时间。

### 为什么不能只靠文件名

- 不同手机、导出工具、聊天软件转存后的命名不稳定。
- 一旦文件名格式变化，解析规则就容易失效。

### 实现建议

预处理脚本优先调用 `exiftool` 获取拍摄或创建日期，读不到时回退到文件 `mtime`，并把文件复制到：

```text
<output-dir>/YYYY-MM-DD/<original-filename>
```

Go 服务仍可调用 `ffprobe` 获取：

- creation time
- duration
- width
- height

如果 `ffprobe` 读不到时间，再回退到文件名和 `mtime`。

### 预处理脚本

脚本保存到：

```text
shafurui-backend/scripts/group-videos-by-date.sh
```

使用示例：

```bash
cd shafurui-backend
./scripts/group-videos-by-date.sh ~/video-export-flat ~/video-export-grouped
```

命令说明：

- `~/video-export-flat` 是手机导出的扁平视频目录。
- `~/video-export-grouped` 是整理后的中间目录。
- 脚本会递归查找 `mp4`、`mov`、`m4v`、`webm` 文件。
- 已存在的目标文件会跳过，避免重复复制。

建议不要直接在手机导出的原始目录里移动文件。保留原始目录，生成一个中间目录，更容易回滚和重新处理。

## MOV 到 MP4 预处理策略

手机导出的视频经常是 `mov`，但 PC Web 的 `<video>` 对 `mp4` 的兼容性通常更稳定，因此建议在上传前生成 `mp4` 播放版本。这里的优先方案不是转码，而是用 `ffmpeg` 重新封装：

```bash
ffmpeg -i input.MOV \
  -map 0:v:0 \
  -map 0:a:0? \
  -c copy \
  -movflags +faststart \
  -map_metadata 0 \
  output.mp4
```

这个命令的关键点：

- `-c copy` 表示只换容器，不重新编码，视频和音频码流不变，因此不会降低画质。
- `-movflags +faststart` 会把 MP4 索引提前，更适合 Web 边下边播。
- `-map 0:v:0` 和 `-map 0:a:0?` 只保留主视频流和可选主音频流，跳过 iPhone 常见的 `mebx` 数据流。
- `-map_metadata 0` 会尽量保留原视频元数据，便于后续用 `ffprobe` 读取创建时间。

以 `IMG_4466.MOV` 为例，它的视频编码是 `H.264 High`，像素格式是 `yuv420p`，音频是 `AAC LC`，本身已经是浏览器友好的编码组合。准确处理方式是 `MOV -> MP4` 无损重新封装，不需要转成 `libx264`。只有遇到 HEVC、异常音频编码或浏览器无法播放的文件时，才考虑额外转码为 `H.264 + AAC`。

### 预处理脚本

脚本保存到：

```text
shafurui-backend/scripts/remux-mov-to-mp4.sh
```

使用示例：

```bash
cd shafurui-backend
./scripts/remux-mov-to-mp4.sh ~/video-export-grouped
```

命令说明：

- `~/video-export-grouped` 是已经按日期整理后的目录。
- 脚本会递归查找 `mov`、`m4v` 文件。
- 输出文件和源文件同目录、同名，但扩展名为 `mp4`。
- 如果目标 `mp4` 已存在，会直接跳过。
- 脚本只做 `-c copy` 无损重新封装，不降低视频质量。
- 成功生成 `mp4` 后会删除中间目录里的 `mov` / `m4v` 副本，避免 API 扫描和页面展示出现重复视频；原始素材仍保留在手机导出的原始目录中。

## 封面图预处理策略

封面图也建议在上传前用脚本生成，而不是由 Go API 触发。这样可以避免线上接口执行大量 `ffmpeg` 任务，也能让静态目录在同步完成后立即具备完整的播放和封面资源。

### 推荐封面命名规则

```text
2026-07-01/a.mp4  ->  2026-07-01/a.jpg
2026-07-01/b.mp4  ->  2026-07-01/b.jpg
```

### 预处理脚本

脚本保存到：

```text
shafurui-backend/scripts/generate-video-covers.sh
```

使用示例：

```bash
cd shafurui-backend
./scripts/generate-video-covers.sh ~/video-export-grouped
```

命令说明：

- `~/video-export-grouped` 是已经按日期整理后的目录。
- 脚本会递归查找视频文件。
- 如果同目录下已经存在同名 `jpg`，会直接跳过。
- 如果封面不存在，会用 `ffmpeg` 从视频第 1 秒附近截取一帧生成 `jpg`。

推荐完整本地预处理流程：

```bash
cd shafurui-backend
./scripts/group-videos-by-date.sh ~/video-export-flat ~/video-export-grouped
./scripts/remux-mov-to-mp4.sh ~/video-export-grouped
./scripts/generate-video-covers.sh ~/video-export-grouped
rsync -av --progress ~/video-export-grouped/ user@server:/data/www/site/videos/
```

## Go API 设计建议

### 角色定位

Go API 不负责播放视频，不代理视频内容，也不负责生成封面图。它只负责读取配置的视频目录、扫描文件、读取必要元数据，并返回列表数据。视频文件和封面图都由 nginx 或 Gin 静态文件服务以配置的访问前缀提供。

当前后端使用以下配置项控制视频来源和返回 URL：

```yaml
video_dir_path: "./public/video-export-grouped"
video_base_url: "http://localhost:6305/public/video-export-grouped"
```

- `video_dir_path` 是 API 扫描的本地视频根目录。
- `video_base_url` 是返回给前端的静态资源访问前缀。
- API 不返回本地绝对路径，只返回相对路径和基于 `video_base_url` 拼接出的 `url` / `coverUrl`。

### 推荐接口

#### `GET /api/health`

用于健康检查。

返回示例：

```json
{
  "ok": true
}
```

#### `GET /api/video`

返回视频分组列表。

推荐返回结构：

```json
{
  "groups": [
    {
      "date": "2026-07-01",
      "items": [
        {
          "id": "2026/07/a.mp4",
          "filename": "a.mp4",
          "relativePath": "2026/07/a.mp4",
          "url": "http://localhost:6305/public/video-export-grouped/2026/07/a.mp4",
          "coverUrl": "http://localhost:6305/public/video-export-grouped/2026/07/a.jpg",
          "shotAt": "2026-07-01T10:23:11+08:00",
          "groupDate": "2026-07-01",
          "durationSec": 12.4,
          "width": 1920,
          "height": 1080,
          "sizeBytes": 12345678,
          "mtime": "2026-07-01T11:00:00+08:00"
        }
      ]
    }
  ]
}
```

### 扫描逻辑

- 从 `video_dir_path` 读取并递归扫描指定视频根目录
- 过滤非视频文件
- 支持扩展名白名单：
  - `mp4`
  - `mov`
  - `m4v`
  - `webm`
- 对每个视频读取基础文件信息
- 调用 `ffprobe` 读取元数据
- 根据时间优先级生成 `shotAt`
- 按 `shotAt` 倒序排列
- 再按 `groupDate` 分组输出
- 根据视频相对路径推导同名 `jpg` 路径，并用 `video_base_url` 拼接返回 `url` 和 `coverUrl`

### 错误处理建议

- 单个文件元数据读取失败时跳过该字段，不要让整个接口失败
- 如果扫描目录失败，接口返回明确错误信息
- 如果某些文件损坏，保留基本文件信息并标记元数据缺失
- 如果封面图不存在，仍返回视频项，前端降级显示文件名或视频首帧区域

## 前端页面设计建议

### 页面形态

PC 端优先，做成纵向相册流，不做复杂瀑布流。原因是视频卡片高度差异较大，瀑布流在视频场景里不一定更好维护。

### 页面结构

- 顶部：
  - 页面标题
  - 视频总数
  - 可选的日期导航
- 主体：
  - 按日期分组
  - 每组显示日期标题
  - 组内展示视频卡片列表

### 视频卡片建议信息

- 封面图或视频首帧区域
- 文件名
- 拍摄时间
- 时长
- 分辨率

### 交互建议

- 点击卡片后原地播放
- 支持浏览器原生视频控件
- 页面初版不做自动播放，避免带宽和性能问题
- 数据多时增加懒加载或分页
- 页面只消费现有封面图，不负责触发封面生成逻辑

## 关键优化点

### 1. 不建议长期坚持“每次请求实时扫目录”

你现在倾向实时扫描，这个可以作为 v1，但不建议长期保留。

原因：

- 视频多了以后，目录遍历本身就会慢
- `ffprobe` 是最重的步骤
- 每次打开页面都重复做同样工作，浪费 CPU 和 I/O

推荐后续升级为：

- 启动时扫描一次并缓存到内存
- 或扫描后写入一个 JSON 索引文件
- `rsync` 完成后通过 webhook 或命令触发刷新

### 2. 预处理封面图

如果只有播放器缩略区域，没有预生成封面，页面观感会比较弱。这里建议在上传前用 `shafurui-backend/scripts/generate-video-covers.sh` 生成封面，而不是在 Go API 中生成。

推荐方案：

- 在本地或维护机上执行封面预处理脚本
- 用 `ffmpeg` 读取视频第 1 秒附近的一帧
- 生成与视频同名的 `jpg`
- 只处理当前没有封面图的视频，已存在则跳过
- 和视频文件一起通过 `rsync` 上传

这样前端卡片可以先显示封面图，再点击播放。

推荐命令示例：

```bash
cd shafurui-backend
./scripts/generate-video-covers.sh ~/video-export-grouped
```

命令作用说明：

- `cd shafurui-backend` 进入后端子项目目录。
- `./scripts/generate-video-covers.sh` 执行封面预处理脚本。
- `~/video-export-grouped` 是已经按日期整理后的目录。

### 3. 增加基础访问控制

虽然是私有自用，也建议至少做一层保护：

- nginx basic auth
- 内网访问
- VPN 后访问
- 反向代理层限制来源 IP

原因很直接：家庭视频不适合裸露在公网。

### 4. 规范同步命令

`rsync` 最好固定参数，重点是：

- 保留时间戳
- 保留目录结构
- 支持增量同步
- 可选删除远端已不存在文件

示例思路：

```bash
rsync -av --progress /local/export/ user@server:/data/www/site/videos/
```

其中：

- `-a` 表示归档模式，会尽量保留时间、权限、目录结构
- `-v` 表示显示详细信息
- `--progress` 表示显示传输进度

如果后续需要严格镜像，可再评估是否启用 `--delete`，但这个参数有误删风险，不能默认直接上。

### 5. URL 和路径映射要固定

建议明确约定：

- 文件系统根目录通过 `video_dir_path` 配置，例如 `/data/www/site/videos`
- URL 前缀通过 `video_base_url` 配置，例如 `https://example.com/videos`

这样 Go API 可以稳定生成：

```text
relativePath -> 2026/07/a.mp4
url -> https://example.com/videos/2026/07/a.mp4
```

不要在 API 中返回本地绝对路径。

## 需要完善的地方

### 1. 是否需要搜索和筛选

如果只是自己偶尔看，v1 可以不做。

如果视频数量会持续增加，建议后续增加：

- 按月份筛选
- 按年份快速跳转
- 按文件名关键词搜索

### 2. 是否需要缩略索引缓存

如果后续加入封面图，建议同时维护一个简单索引，避免前端频繁试探文件是否存在。v1 也可以先在 `GET /api/video` 中直接返回 `coverUrl`。

### 3. 是否需要去重机制

手机导出和多次同步后，可能有重复视频。

后续可增加简单去重策略：

- 基于相对路径
- 基于文件大小 + 时长
- 基于 hash

其中 hash 最准确，但成本最高。

### 4. 是否需要日志和观测

建议至少记录：

- 扫描耗时
- 文件数量
- 元数据提取失败数量
- API 请求耗时

这样后续你能快速判断瓶颈是在磁盘、`ffprobe` 还是前端渲染。

## 建议的实施顺序

### v1

- 建 nginx 静态目录
- 增加视频日期分组预处理脚本
- 增加视频封面图预处理脚本
- 定好“预处理目录 -> `rsync` -> 服务器静态目录”的同步路径
- 写 Go API 按配置扫描视频目录并返回分组结果
- 做 PC Web 页面按日期展示和播放

### v1.1

- 增加基础认证
- 增加扫描耗时日志
- 在列表接口中补充更完整的封面状态字段，例如 `coverExists`

### v2

- 改成扫描缓存索引
- 支持手动刷新索引
- 增加月份导航和筛选

## 验收标准

- 页面能按日期分组展示服务器目录中的视频
- 视频可以通过页面直接播放
- 已生成封面图的视频能在列表中优先显示封面
- 分组预处理脚本能把扁平视频目录整理为按日期分组的目录
- 封面预处理脚本只会处理缺失封面的文件，已存在封面不会重复生成
- 元数据存在时按真实拍摄时间分组
- 元数据缺失时能自动回退
- 新增同步的视频能在合理时间内出现在页面中
- 1000 个级别视频时页面仍可用，接口响应时间可接受

## 最终建议

这个方案适合作为一个轻量、私有、自维护的视频相册系统来做，技术路线没有问题。真正需要提前避免的不是“能不能做”，而是两个后期问题：

- 不要长期依赖“每次请求实时扫目录 + 每个文件都现查元数据”
- 不要忽略上传前预处理、封面图生成机制和访问控制

如果你后续要进入实现阶段，建议按下面的技术拆分推进：

- Shell：日期分组预处理、封面图预处理、同步前资源整理
- Go：按配置读取目录、元数据提取、分组 API、按配置前缀返回视频和封面 URL
- nginx：静态文件、访问控制、缓存
- 前端：分组相册页、卡片布局、播放器交互

这样能最快做出可用版本，也便于后续逐步优化。
